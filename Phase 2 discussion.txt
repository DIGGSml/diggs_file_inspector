Phase 1 summary — what was built:
* DIGGS_SF — a Map<sfType → SF[]> that is the unified registry of all sampling features. Populated dynamically by walking every <diggs:samplingFeature> wrapper and reading its first element child's localName. No fixed type list — every SF type present in the file is discovered automatically.
* Legacy aliases DIGGS_BH, DIGGS_CPT, DIGGS_WELL remain live references into DIGGS_SF for backward compatibility with existing table/detail/map code.
* SF_TYPE_CONFIG — a const object keyed by SF element localName covering all 17 concrete DIGGS 3.0 SF types. Each entry has tabLabel, mapGeom (point / line / polygon / installation / none), depthPath, and parentPath. No purposePath — purpose is found dynamically by scanning child elements for any localName matching /purpose$/i.
* SF_DETAIL_DESCRIPTORS — empty stub, ready for Phase 2.
* parseSamplingFeature() extended to capture on every SF object: purpose, linearExtentLatLngs, linearExtentSrsName, linearExtentSrsDim, linearExtentLen, linearExtentLenUom, parentRef, childRefs.
* childRefs back-fill — after all SFs are parsed, each installation's parentRef is used to push its id into the parent's childRefs[].
* enrichElevUnits() updated to iterate DIGGS_SF.values() instead of three hard-coded arrays.
* resetToLoad() clears DIGGS_SF and all aliases.
Key architectural notes for Phase 2:
* mapGeom: 'point' — Borehole, Sounding, TrialPit, Station. Render from referencePoint as markers. Borehole/Sounding/TrialPit are geometrically linear (they have centerLine) but are plotted as point markers on a plan-view map.
* mapGeom: 'line' — GP_Trackline, Transect, GroutTrenchCutoffWall. Render from linearExtentLatLngs as polylines. Coordinates are in native CRS (linearExtentSrsName); need CRS conversion for Leaflet.
* mapGeom: 'polygon' — PlanarSamplingFeature, GP_ArealSurvey, TrenchWall, VolumetricSamplingFeature. Render from featureExtent as polygon borders, no fill.
* mapGeom: 'installation' — Well, SteelHPile, SteelPipePile, ConcretePile, TimberPile. No independent map marker. Linked to parent via parentRef; parent's marker pulses on hover/select.
* mapGeom: 'none' — GP_MultiTrack. No own geometry; rendered via member GP_Trackline refs.
* linearExtentLen / linearExtentLenUom come from the optional convenience element and may be null. A future enrichLinearExtentLengths() async pass (modelled on enrichElevUnits()) should compute length from vertices when absent — but this is deferred and not part of Phase 2 scope. Do not implement it in Phase 2 without being asked.

2. Map rendering for line and polygon geometry types (point markers already work for BH/CPT/Well).
3. GP_MultiTrack grouping in table.
4. Project geometry rendering.
5. assoc-panel population.

Layout
4-pane layout replacing the current 3-pane:

Content within Detail and Assoc is horizontally scrollable per-section (via overflow-x: auto on section wrappers, not the whole panel), so panel chrome stays anchored.

SF Assoc Panel is always present as a fixed-width column — it does not slide in or collapse. This keeps Detail's position and width stable when data loads. It shows quiet/muted empty content until an SF is selected and associated data is triggered. It has a minimum width.

Project Selector
Dropdown only — no pills. Cleaner, especially in the narrower Summary pane.
"Project" label appears above the dropdown so it's clear what it controls.
Dropdown has a visible styled background and border (accent color), making it clearly interactive against the dark theme. Browser-native arrow must also be visible.
"All Projects" is always the last option in the dropdown, no info button.
No "No Project" or "Unassigned" entry — by schema, every SF has a projectRef. If a projectRef href doesn't resolve to a Project element in the file, the href fragment itself is used as the label and no info button is shown. If it does resolve, the Project's gml:name is the label and the info button is enabled.
SFs with multiple projectRef elements appear under all matching project filter views.
The existing synthetic "No Project Identified" entry is retired.
SF Summary Table
Unified columns across all SF types: Name | Depth/Length | Date | Parent Feature | Installations | Purpose

No lat/lon or elevation columns — spatial association is better communicated via map highlighting; these columns are unnecessary and removing them allows a narrower summary pane.

Parent Feature (installations only): clickable mini-link that switches to the appropriate SF tab and selects the parent row and map marker.

SF_TYPE_CONFIG

Coordinate display — Option A (native CRS):

Show the raw pre-transform coordinates from the file, not the WGS84-converted values used for map placement.
